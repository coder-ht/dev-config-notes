# Cube2 项目经验总结

## 后端实现

- 场景：新增或修改 Cube2 后端能力。
- 做法：先按业务入口定位应用，优先复用 `packages/platform`、已有 client 和服务；Controller、Service 按业务边界保持职责清晰。
- 注意：不要在循环中访问数据库或外部 client，不要把环境配置、凭据和运行态数据写入规则或经验。

## AI Flow 与接口调试

- 场景：AI Flow、工作流或平台接口返回结构不稳定。
- 做法：先确认真实请求路径、Header、响应包装层和业务结果字段，再编写解析或验证逻辑。
- 注意：不能直接把整个数组或包装对象当作业务结果；401/403 优先检查当前有效鉴权信息。

## 定时任务与测试造数

- 场景：定时任务、租户字典、工作流配置或测试数据交付。
- 做法：代码完成后继续核对租户字典、工作流 ID 和运行前置配置；批量造数保留唯一 ID，验证关联完整性并准备逻辑回滚。
- 注意：具体应用表名、flow ID、tenant、测试地址和业务字段只在当前任务中核实，不沉淀为通用经验。

## 测试环境菜单与隐患治理造数

- 场景：测试环境管理端菜单或隐患治理根因分析数据需要处理。
- 做法：菜单更新时先完整读取现有字段，只修改目标 `enabled`；隐患治理“已关闭”按业务查询窗口和历史闭环记录实时派生，造数使用新组合并核对关联数据。
- 注意：测试接口使用当前有效鉴权；不沉淀账号、Bearer、Cookie、验证码或完整测试地址；正式环境补录保留审计痕迹并按唯一 ID 做回滚核对。
