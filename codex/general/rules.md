# 通用规则

本文件记录可迁移到其他 Codex 环境的通用协作和编码规则。

## Codex 环境变量强制要求

- Codex 必须配置并读取环境变量 `CODEX_WORK_ENV` 后才能正常执行任务。
- `CODEX_WORK_ENV` 用于判断当前运行环境，并决定需要加载哪些环境规则和环境经验文件。
- 当前支持的环境值包括 `ghy` 和 `home`。
- 每次执行用户任务前，必须先读取 `CODEX_WORK_ENV`。
- 若 `CODEX_WORK_ENV` 为空、缺失或不是已支持的环境值，必须停止执行具体任务，先提示用户配置环境变量。
- 环境变量缺失时，只允许说明需要配置什么，不允许继续执行代码修改、接口调试、规则同步、规则备份、提交或推送等实际操作。
- 配置示例：`export CODEX_WORK_ENV=ghy` 或 `export CODEX_WORK_ENV=home`。
- 只有确认 `CODEX_WORK_ENV` 有效后，才能继续读取通用规则、通用经验、对应环境规则和对应环境经验。

## 经验总结高优先级规则

- 经验总结是任务执行闭环和完成条件，不是可选的收尾动作；除更高层级指令、用户当前明确要求和安全边界外，本节优先于普通执行流程。
- 任务开始前必须读取并检索通用经验与当前环境经验，在执行计划中明确说明命中的经验或“未发现相似经验”。
- 任务执行结束后、给出最终结论前，必须再次检索经验文件，判断本次应优化已有经验、追加新经验，还是确认无需变更。
- 任务尚未完成经验复查与处理时，不得宣称任务已全部完成；最终回复必须明确说明经验文件的处理结果。
- 不得因任务简单、改动较少、已经提交或已经推送而跳过经验总结；经验内容仍应保持简短，只记录可复用的判断依据、操作边界和踩坑结论。
- 写入经验时必须遵守经验归属和敏感信息边界，不得为了完成经验总结而扩大代码、配置、提交或推送范围。

## 协作流程与经验归属

- 每次开始执行用户任务前，先说明准备做哪些事情。
- 每次开始执行用户任务前，必须读取当前环境中的规则文件和经验总结文件；不得只凭记忆或历史上下文跳过读取。
- 主动检索通用经验与当前环境经验中是否存在与当前任务相似的经验总结。
- 检索经验时，必须同时覆盖通用经验与当前环境经验；若任务明确涉及某个项目、路径、命令、报错或工作流，应使用这些关键词进行检索。
- 若应读取的经验总结文件不存在，先创建该文件，再继续列计划。
- 每次开始执行实际修改、远端操作、提交、推送、部署、规则写入或可能影响环境状态的命令前，必须先列出简明执行计划，并等待用户确认后才能继续。
- 只有用户在同一轮请求中明确表达“直接执行”“无需确认”“马上执行”等免确认意图时，才能在说明计划后继续执行；普通的任务描述、问题描述或修复请求不视为免确认。
- 执行前计划必须包含：本次目标、将检查或修改的关键文件/配置、命中的可参考经验或“未发现相似经验”、预计验证方式、是否需要暂存/提交/推送。
- 若命中可参考经验，计划中必须说明参考了哪类经验以及本次如何使用；若未命中，必须明确写“未发现相似经验”，不要含糊带过。
- 经验参考只能用于缩小排查范围或复用已验证流程，不能替代读取当前真实文件、当前命令输出、当前配置或当前接口响应。
- 每次执行完用户任务后，必须说明本次做了什么、改了哪些关键文件或配置、是否完成验证、验证命令或验证方式、是否存在未暂存或需要用户关注的事项。
- 执行后说明必须区分“已完成的改动”“已验证的结果”“未验证或跳过验证的原因”“遗留风险/用户需关注事项”，不能只给笼统结论。
- 每次执行完用户任务后，再次读取通用经验与当前环境经验，检索是否已有类似经验。
- 若已有类似经验，判断本次是否提供新的可复用信息；有新增价值时优化原条目，无新增价值时说明已检查无需变更。
- 若没有类似经验，追加本次任务的简明经验总结。
- 经验总结必须记录可复用的规则、判断依据、命令边界或踩坑结论。
- 经验总结必须足够短，优先写“场景 / 做法 / 注意”三类信息，避免把执行过程全文搬进经验文件。
- 写入经验前必须判断归属：所有环境都适用的经验写入通用经验文件；只和某个环境的路径、项目、配置、账号、机器状态相关的经验写入对应环境经验文件。
- 不确定经验归属时，先询问用户，不要随意写入通用经验。
- 最终回复必须说明本次对经验总结文件的处理结果：已优化、已追加或已检查无需变更。
- 新编写或修改的代码完成后，默认放到 git 暂存区。
- 若存在环境配置、临时文件或与任务无关的改动，应保持未暂存并说明。

## Codex 规则同步与备份

- 当用户说“同步 Codex 规则”“从仓库同步规则”“把仓库规则同步到本机”或类似表达时，含义是：以当前规则仓库为来源，把仓库内 `codex/` 目录中的规则/经验文件同步到当前本机 Codex 配置中。
- 当用户说“备份 Codex 规则”“把本机 Codex 规则放到仓库”“备份当前本机规则”或类似表达时，含义是：以当前本机 Codex 配置为来源，把本机规则相关文件整理到当前规则仓库的 `codex/` 目录中，然后提交并推送。
- 执行“同步”或“备份”前，必须先确认方向，不能混淆：同步是“规则仓库 -> 本机 Codex 配置”，备份是“本机 Codex 配置 -> 规则仓库”。
- 执行前必须读取当前系统环境变量 `CODEX_WORK_ENV`，判断当前环境，并按环境选择对应目录：通用规则/经验使用 `codex/general/`，环境规则/经验使用 `codex/envs/<env>/`，路由入口使用 `codex/AGENTS.md`。
- 执行“同步”前，必须先进入当前环境的规则仓库根目录拉取最新代码；优先使用 `git pull --ff-only`，若无法快进或拉取失败，必须停止同步并说明原因，不能用旧仓库内容覆盖本机配置。
- 执行“同步”前若规则仓库存在未提交改动，必须先说明改动范围；只有确认这些改动不会被拉取覆盖或用户明确要求继续时，才能继续拉取与同步。
- 同步到本机时，`~/.codex/AGENTS.md` 应使用仓库中的 `codex/AGENTS.md` 路由逻辑；不要把所有具体规则堆回 `~/.codex/AGENTS.md`。
- 执行“备份”前，必须先进入当前环境的规则仓库根目录检查 `git status`，说明已有未提交或未跟踪改动是否属于本次备份。
- 执行“备份”写入仓库文件前，必须先拉取远端最新代码；优先使用 `git pull --ff-only`，若无法快进或拉取失败，必须停止备份并说明原因，不能跳过拉取后直接提交或推送。
- 备份到仓库时，按用户要求决定备份范围，并在执行后说明实际处理了哪些文件。
- 备份到仓库后，必须暂存、提交并推送相关仓库文件；若推送被远端拒绝，必须先拉取并整合远端更新，再重新验证后推送。
- 同步到本机后，必须说明写入或覆盖了哪些本机文件。
- 最终回复必须说明：本次执行的是“同步”还是“备份”、来源路径和目标路径、处理了哪些规则/经验文件；若是备份，说明提交号和推送结果；若是同步，说明本机 `~/.codex/AGENTS.md` 是否已更新为路由入口。
- 如果用户只说“同步一下”或“备份一下”但方向不明确，必须先询问用户确认方向。

## 应用安装与经验沉淀

- 当用户要求安装、迁移或升级应用时，执行前必须在当前环境的 `dev-config-notes` 仓库中检索通用经验与当前环境经验，使用应用名称、安装方式、目标路径和操作系统等关键词。
- 若命中可参考经验，应在执行计划中说明将如何参考；经验不能替代对当前安装包、安装目录、配置和验证结果的检查。
- 若未命中经验，完成安装并验证后，必须将简短、可复用的经验追加到正确归属的 `experience.md`：通用流程写入 `codex/general/experience.md`，环境专属路径、配置或机器状态写入 `codex/envs/<env>/experience.md`。
- 追加安装经验后，只暂存本次经验相关文件并创建 Git 提交；不得将已有的无关修改混入该提交。用户另有指示时，按用户指示处理提交与推送。

## 基本编码原则

- 编写代码的第一重要准则：在不影响业务语义、接口契约、运行逻辑和可维护性的前提下，代码必须尽量精简、直接、无冗余。
- 不要引入不必要的抽象、包装、分支、临时变量、重复字段或重复逻辑。
- 优先复用当前项目已有的基础能力、工具类、模型、常量、客户端和标准库。
- 不要重复封装通用判空、字符串处理、集合/Map 处理、分页转换、JSON 处理、异常处理等基础工具，除非确实存在业务边界内的专用语义。

## 前端项目 Git 提交流程

- 对前端应用执行 `ga&gc` 或等价的暂存并提交流程前，必须先检查并升级该应用自身的 `package.json` 版本号。
- 版本号升级应只作用于本次要提交的前端应用包，不要顺手修改无关包版本。
- 若本次提交不适合升级版本号，必须在执行 `ga&gc` 前向用户说明原因并确认。

## 领域建模

- 对于状态、类型、来源、阶段、级别等有限值域，能使用枚举的统一使用枚举。
- 不要在业务代码、接口设计或表结构设计中散落硬编码字符串。
- 编写代码时，不要在业务代码里硬编码可复用字符串。
- 若确实需要字符串常量，应按业务边界新增或复用常量类，或通过配置/字典承载，并通过统一引用使用。

## Java 后台职责拆分

- 编写 Java 代码时，优先按面向对象思想拆分职责。
- 类应保持单一职责。
- 方法应归属于合适的领域对象、服务对象、转换器、配置解析器或工具类中。
- 不应把无关逻辑集中堆叠在同一个类里。
- 对于功能相同或职责相近的类，应按业务域或技术职责单独建包统一管理。
- 抽象和拆包必须基于真实复用、复杂度隔离或边界清晰的需要，避免过度设计。

## Service 设计

- Service 必须按业务域、资源对象和职责边界拆分。
- 不要把不同职责堆到同一个 Service。
- 新增或重构 Service 时，先定义 `service` 包下的业务接口，再在 `service/impl` 包下提供对应 `*Impl` 实现类。
- Controller 应依赖业务接口而不是实现类。
- Service 实现应保持流程编排简洁。
- 复杂的参数校验、VO/DTO 转换、外部数据补全、配置解析、指标/趋势组装等逻辑，应按职责抽到专用组件、Assembler、Validator、Provider、Manager 或 Helper 中。
- 辅助组件也必须按业务域或技术职责继续分包。
- 包名要表达所属领域，类名要表达具体职责。
- 优先使用清晰后缀，如 `Assembler`、`Validator`、`Provider`、`Manager`。
- 不要把大量语义不明的 `*Support` 类堆在同一个根包下。

## Controller 设计

- 编写后台接口时，应根据业务域、资源对象和职责边界拆分不同 Controller。
- 不要把无关或弱相关接口全部堆到同一个 Controller。
- 新增接口前先判断是否已有合适 Controller；若没有，再按业务边界新建。
- Controller 类和接口方法需要添加简洁、合理的注释，说明接口职责和关键行为。
- 接口方法、请求参数类、响应 VO/DTO 字段以及必要的业务逻辑都需要添加清晰注释。
- 避免空泛或重复代码含义的注释。

## 数据与外部调用

- 不要在循环中调用数据库。
- 不要在循环中调用外部或内部 client。
- 数据库操作包括 mapper/service/repository 的 select、insert、update、delete、save/remove 等。
- 应改为批量查询、批量写入或一次性 client 调用。
- 若确实必须在循环中调用，需要先向用户确认，并说明为什么无法批量化、影响范围和风险。
- 编写跨模块或跨服务能力时，若已有 client 或服务可复用，应优先复用现有能力。
- 不要新写重复 client、重复封装接口或重复建设数据源。
- 如果确实需要新增，应先告知用户新增原因、现有能力不能满足的点以及影响范围。

## 领域数据来源

- 编写涉及外部领域对象展示、详情补全、实时状态或历史数据的代码时，应优先从对应领域的权威服务、已有 client 或配置中心获取。
- 业务模块不要复制外部数据源、保存不必要的展示快照或跨库直连外部领域表。

## 异常处理

- 编写业务代码需要抛异常时，优先使用项目统一异常体系和既有错误码/消息机制。
- 不要随意抛裸 `RuntimeException`、`IllegalArgumentException` 或自造异常体系，除非有明确框架或边界原因。

## 日期时间

- 编写涉及业务日期时间的代码时，应使用项目统一的日期时间类型和数据库字段类型。
- 不要用字符串、纯日期或时间戳随意承载需要精确到时间的业务时间。

## 日志

- 应在任务启动/结束、外部系统调用、配置读取、批量处理、关键状态分支和异常捕获处添加必要日志。
- 日志内容应包含业务标识、租户、数量、配置编码、任务名、外部调用目标等排查字段。
- 避免在循环中打印大量逐条日志。
- 批量场景优先记录汇总数量和失败摘要。
