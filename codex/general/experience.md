# 通用经验总结

本文档记录所有环境都适用的 Codex 协作、编码、规则维护和仓库操作经验。

## 2026-07-16 根据用户水平调整解释

- 场景：同一问题面向不同知识背景的用户，需要采用不同的解释深度和表达方式。
- 做法：任务开始读取 `codex/user/technical-levels.md`，先识别当前问题的主要技术和前置技术，再按各自评分分别决定解释深度，不使用综合分或简单平均分；称呼只使用“技术主题 + 英雄联盟排位等级 + 当前分数”，任务结束仅在出现有效新证据时渐进更新档案。
- 注意：称呼不添加额外角色或修饰词；应把 1–100 分均分映射到坚韧黑铁至最强王者十档，低置信度标注“暂定”。称呼和解释深度都不能固化用户身份或牺牲答案准确性。

## 2026-07-16 技术评分与阶段学习计划

- 场景：用户要求区分真实测试结果与早期协作推断，并在提升某项技术评分时持续出题。
- 做法：未完成专项测试的技术标记为“未测试”；已评分技术在 `codex/user/learning-plans/` 下维护独立计划，评分档案只引用计划文件；计划只覆盖当前英雄联盟段位，后续校准题从计划的待加强内容和验证练习中选择。
- 注意：不能把未测试技术的推断分数当作正式水平，也不能提前用高段位内容出题；评分或答题产生新证据后必须同步更新评分档案和对应计划文件。

## 2026-07-16 技术学习经验持续校准

- 场景：学习计划需要反映用户真实答题过程，而不是只列出预设知识点。
- 做法：为每项已评分技术建立独立学习经验文件，记录错误项、纠正状态、已学习内容、已回答问题、复测证据和掌握进度；每次回答后追加证据，答对或迁移应用后再更新纠正状态。
- 注意：不删除历史错误，不把“听过解释”当作“已掌握”；学习经验、学习计划和评分档案产生新证据时必须同步更新，未测试技术不创建文件。

## 2026-07-16 评分与复测机制优化

- 场景：单一总分和一次答对容易高估掌握程度，学习计划也可能因重复原题而产生假进步。
- 做法：按子能力记录证据等级，使用“初步掌握—基本掌握—稳定掌握”的复测门槛；采用即时变式题、延迟复测和迁移题，并将评分调整与置信度分开维护。
- 注意：不能把复述解释当作掌握，不能连续重复原题；进入新段位时归档旧计划并继承未纠正错误，用户可纠正或要求删除证据。

## 2026-07-16 学习机制长期维护优化

- 场景：评分机制长期运行后，容易混淆学习、复测和正式评分，也可能因跨技术前置知识、重复题目或历史记录膨胀而失真。
- 做法：区分三种答题模式，按子能力权重和部分得分记录证据；学习计划记录目标、依赖、版本和复测日期，进入新段位时归档；经验文件限制近期明细并长期保留错误摘要。
- 注意：不同技术的证据不能混算；30 天未复测标记待复习，90 天只降低置信度；用户可以要求只学习、不评分或纠正证据归属。

## 2026-07-13 默认使用 subagent

- 场景：需要统一 Codex 的任务协作方式。
- 做法：在通用规则中约定，任务允许且 subagent 可用时，默认优先委派 subagent；主 agent 负责拆解、协调、验证和汇总。
- 注意：subagent 不可用或不适合任务时，直接由主 agent 执行。

## 经验归属规则

- 所有环境都适用的经验写入本文件。
- 只和某个环境的路径、项目、配置、账号、机器状态相关的经验，写入对应环境目录下的 `experience.md`。
- 不确定归属时，先询问用户，不要随意写入通用经验。

## 2026-07-09 Codex 规则迁移与路由

- 场景：需要把 Codex 全局规则放入仓库，方便不同机器的新 Codex 复用。
- 做法：`AGENTS.md` 只保留环境路由逻辑，具体规则拆到 `codex/general/` 与 `codex/envs/<env>/`。
- 复用：任务开始时先读取 `CODEX_WORK_ENV`，始终加载通用规则/经验，再按环境加载对应规则/经验。
- 注意：写入经验前必须判断归属，通用经验写入 `codex/general/experience.md`，环境专属经验写入对应环境目录。
- 约定：入口 `AGENTS.md` 必须直接读取当前环境 `dev-config-notes` 项目中的规则文件，不读取本机 `~/.codex/codex/` 副本。

## 2026-07-09 Codex 规则同步与备份术语

- 场景：用户要求区分“同步 Codex 规则”和“备份 Codex 规则”的方向。
- 做法：“同步”固定表示规则仓库到本机 Codex 配置，并同时处理入口、规则和经验；“备份”固定表示本机 Codex 配置到规则仓库，并同时处理入口、规则和经验，完成后提交推送。
- 注意：执行前必须确认方向和 `CODEX_WORK_ENV`。

## 2026-07-09 Codex 规则同步与备份前拉取仓库

- 场景：同步或备份 Codex 规则前，规则仓库可能落后于远端。
- 做法：同步和备份都先检查规则仓库状态并执行 `git pull --ff-only`；同步后将仓库中的入口、规则和经验写入本机，备份前再逐文件比较本机内容后写回仓库。
- 注意：无法快进或拉取失败时停止同步/备份；推送被远端拒绝时，先拉取并整合远端更新，再重新验证后推送。

## 2026-07-09 Codex 环境变量强制要求

- 场景：用户要求 Codex 必须配置环境变量后才能正常工作。
- 做法：`CODEX_WORK_ENV` 是必需环境变量，当前支持 `ghy`、`home-windows` 和 `home-wsl`；缺失或未知时必须停止具体任务并提示用户配置。
- 注意：规则维护任务可用于落地这条约束；后续普通任务、同步、备份、提交、推送等都必须先确认 `CODEX_WORK_ENV` 有效。

## 2026-07-09 Codex 执行说明与经验参考强化

- 场景：用户要求加强经验总结、经验参考、执行前计划说明和执行后说明的规则约束。
- 做法：将经验总结提升为通用规则中的独立高优先级章节和任务完成门槛；任务前必须检索并在计划中说明，未找到完整覆盖本次场景的经验时，任务结束前必须新增；任务后必须复查、处理经验并在最终回复中说明文件和条目。
- 注意：经验只能辅助缩小范围，不能替代读取当前真实文件、命令输出、配置或接口响应；不得因任务简单、仅排查、失败、中断、未改代码、已经提交或已经推送而跳过，只要产生新的判断方法、踩坑、边界或验证方式就必须记录。

## 2026-07-09 接口请求信息记录边界

- 场景：用户提供测试环境 curl，希望记录接口地址、请求头和 token 信息。
- 做法：按用户要求记录 Host、路径、Header 名称、tenant、terminal、content-type、Referer、Authorization、Cookie 和 token 相关信息。
- 注意：记录测试环境请求信息时，优先保留接口复用所需字段。

## 2026-07-09 前端项目 ga&gc 版本号规则

- 场景：对前端应用执行 `ga&gc` 或等价暂存提交流程。
- 做法：提交前先检查并升级该应用自身 `package.json` 的版本号。
- 注意：只升级本次提交对应应用包；不适合升级时，先向用户说明原因并确认。

## 2026-07-09 执行前计划确认强化

- 场景：用户要求加强任务执行前的计划告知与确认约束。
- 做法：实际修改、远端操作、提交、推送、部署、规则写入或影响环境状态的命令前，先列计划并等待用户确认。
- 注意：只有用户同轮明确说“直接执行”“无需确认”“马上执行”等，才可在说明计划后继续。

## 2026-07-10 Codex 全局免确认与全权限配置

- 场景：用户希望后续 Codex 会话无需逐条确认，且不受沙箱限制。
- 做法：在全局 `~/.codex/config.toml` 设置 `approval_policy = "never"` 与 `sandbox_mode = "danger-full-access"`，并用 `codex.cmd --strict-config --version` 校验。
- 注意：该组合允许自动执行未受沙箱保护的命令，仅应在用户明确授权且环境本身有额外防护时使用。

## 2026-07-10 应用安装经验沉淀

- 场景：用户要求安装、迁移或升级应用。
- 做法：执行前检索当前 `dev-config-notes` 仓库的通用和环境经验；未命中时，在安装验证后将经验写入正确归属文件，并仅提交本次经验相关文件。
- 注意：环境专属安装目录和机器配置不能写入通用经验；已有无关修改不得混入经验提交。

## 2026-07-13 Codex 规则与经验同步备份的副本校验

- 场景：同步或备份本机 Codex 入口、规则和经验时，本机副本可能只是旧的历史副本。
- 做法：同步以仓库为源覆盖本机；备份先逐文件比较，只有本机内容确实更新时才写回仓库。
- 注意：不能用旧副本覆盖仓库中的新内容；无实际差异时不创建空备份提交。

## 2026-07-14 项目级规则按仓库根路径路由

- 场景：多个项目各自维护规则和经验，需要避免项目规则互相串载。
- 做法：根据完整仓库根路径或其子目录精确映射到 `codex/projects/<project>/`，一次只加载匹配到的一个项目。
- 注意：不要根据当前子目录名称猜测项目；未匹配项目时只加载通用规则和环境规则。

## 2026-07-15 外部系统写入回读验证

- 场景：通过 MCP、API、数据库客户端或其他工具修改外部系统，工具返回成功但实际结构或展示可能不符合预期。
- 做法：写入前读取并保留原结构和关联引用；写入后使用独立读取或查询核对目标对象、关键字段、引用关系和同批次影响范围。
- 注意：成功响应只代表请求被接受，不能替代结果验证；发生 AI 失误时需记录现象、根因、错误做法、正确做法、影响范围和验证方式。

## 2026-07-15 Bash 脚本兼容 sh 调用

- 场景：含 `pipefail`、`[[ ... ]]` 等 Bash 语法的脚本被用户通过 `sh script.sh` 调用，或粘贴缩进导致 shebang 前存在空格。
- 做法：确保 `#!` 从文件首字节开始，并在使用 Bash 专属语法前检测 `BASH_VERSION`、通过 `exec bash "$0" "$@"` 自动切换解释器；使用用户原始调用方式做真实验证。
- 注意：不能简单删除 `pipefail`，否则导出压缩等管道可能掩盖上游失败；`pipefail` 下游提前退出会使上游收到 SIGPIPE 并返回 141，需要让读取端消费完整输入；动态匹配反引号时不要将其直接放进 Shell 双引号，应通过 `printf` 或单引号程序生成字面量，避免触发命令替换。最后同时执行 `bash -n`、用户原始调用方式、产物完整性和校验文件回读。

## 2026-07-16 技术正式评测固定十题且不要求手写实践

- 场景：用户要求控制评测长度，避免围绕同一技术无限追加题目。
- 做法：正式评测固定 10 道题，覆盖主要子能力、结果预测、变式题和综合迁移题；综合能力通过代码阅读、结果预测、错误分析、方案比较和口头解释验证，十题后统一计算评分和置信度。
- 注意：不得要求用户手写实践代码；用户提前结束时标记为未完成，不追加题目凑数，也不把未回答题目当作错误；已有历史评测记录保持不变。

## 2026-07-16 同轮知识点不重复出题

- 场景：用户要求避免同一轮评测中重复出现相同题目或同一知识点。
- 做法：每轮 10 题中同一知识点只考一次；跨轮仅为纠正薄弱项复测，并更换数据、场景或题型。
- 注意：变式复测可以保留，但不能复用原题或把同一知识点拆成多道重复小题。
